01

Prototipos vs. app real

Hay dos cosas distintas con el mismo nombre, MascotApp. No las confundas:

Prototipo · maqueta

Las demos

  • Pantallas de diseño para mostrar la idea
  • Se abren en el navegador con doble clic
  • No guardan datos ni tienen login real
  • Sirven para validar el look & feel con clientes
App · el proyecto

MascotApp

  • El proyecto donde vamos a desarrollar el producto
  • Por ahora casi vacío: arranca el esqueleto (login + base de datos + API)
  • Ya tiene una primera pantalla funcionando de verdad: pacientes
  • Acá se construye todo, de a poco, hasta llegar a producción
🧭

Regla simple: si lo abrís con doble clic y es solo lindo → es un prototipo. Si te pide usuario y contraseña en localhost → es la app real.

02

Qué instalás vos, qué le pedís a Claude

La idea: vos instalás solo 2 programas (tienen instalador, como cualquier app). Todo lo demás (versiones, dependencias, base de datos) se lo pedís a Claude con un mensaje.

Lo que instalás vos — 2 cosas

  • Claude Code — el asistente que hace el trabajo. Es quien instala el resto, levanta la app y programa las features. → docs.anthropic.com/claude-code
  • Docker Desktop — el motor donde corre la base de datos. Lo instalás una vez y lo dejás abierto. → docker.com
  • Una cuenta de GitHub — ahí vive el código. Gratis; solo necesitás el usuario y contraseña.

Lo que le pedís a Claude — todo el resto

Abrí Claude Code y pegale este mensaje. Él se encarga de bajar el código, instalar Node, las dependencias y dejar la base de datos lista:

💬 Pegale esto a Claude

Cloná el repo de MascotApp desde GitHub, instalá todo lo que haga falta (Node 20 con nvm y las dependencias del proyecto) y dejá la base de datos lista para arrancar. Avisame si necesitás que haga login en algún lado.

🔐

Algunas cosas necesitan tu permiso o tu login (entrar a GitHub, por ejemplo). Cuando pase, Claude te lo va a pedir y vos seguís las instrucciones en pantalla — no tenés que adivinar nada.

03

Levantar la app — paso a paso

Tenés dos caminos.

Con Docker Desktop abierto, pedile a Claude que la levante. Cuando termine, te dice la dirección exacta para abrir:

💬 Pegale esto a Claude

Levantá MascotApp en modo desarrollo y decime en qué dirección la abro en el navegador y con qué usuario y contraseña entro.

Si preferís hacerlo vos, es un comando. Primero copiás la config de ejemplo y después lo levantás todo (base + servidor + web):

terminal · en la carpeta del repo
$ cp .env.example .env    # config de ejemplo (editá JWT_SECRET: 32+ caracteres)
$ npm run docker:up      # levanta base + API + web

Cuando ya está corriendo

  1. Abrí el navegador en http://localhost:5173 (si ese número está ocupado, Claude te dirá cuál usar — p. ej. 5174).
  2. Entrá con uno de los usuarios de prueba (abajo).
  3. Andá a Pacientes en el menú: ahí podés cargar, editar y borrar mascotas. Eso ya funciona de verdad.

Si algo no abre, lo más común es que Docker Desktop esté cerrado o que la dirección sea otra. Decile a Claude “la app no abre” y él lo diagnostica.

04

Mostrarla a otra persona (túnel)

Mientras corre en tu compu, la app vive en localhost: solo vos la ves. Un túnel crea un link público temporal para que otra persona —un cliente, un inversor— la abra desde su celular o su compu, sin deployar nada.

Lo más simple y sin cuenta es Cloudflare Tunnel: un comando y te da un link https://…trycloudflare.com. Pero como la app son dos piezas (la web y su API), lo más práctico es pedírselo a Claude, que arma el túnel y deja todo conectado:

💬 Pegale esto a Claude

Exponé MascotApp con un túnel público temporal (tipo Cloudflare) para que pueda mostrársela a alguien de afuera, y pasame el link. Asegurate de que la API también funcione a través del túnel.

Si querés hacerlo a mano

Con la app ya corriendo, en otra terminal:

terminal · túnel a la web 
$ brew install cloudflared                          # una sola vez
$ cloudflared tunnel --url http://localhost:5173    # te imprime el link público
⏳

El link es temporal: funciona solo mientras tu compu esté encendida y la app corriendo. No es un deploy — para algo permanente hace falta publicar la app en un servidor.

🔒

Mientras el túnel está abierto, cualquiera con el link entra. Usalo para demos con datos de prueba, no con información real, y cerralo cuando termines (Ctrl + C en la terminal).

06

De la idea a la funcionalidad

1 · Primero, un PRD corto

Un PRD es un documento breve que deja por escrito qué querés construir y por qué, antes de programar.

Pedile a Claude que lo escriba y lo guarde en el proyecto, así no se pierde:

💬 Pegale esto a Claude

Tengo una idea concreta: una pantalla para gestionar tutores (los dueños de las mascotas). Antes de programar, escribí un PRD corto y fácil de leer —problema, qué hace, qué NO hace, cómo se usa y cuándo está listo— y guardalo en el proyecto en docs/prd/. Después frenamos para que lo revise.

📄

Los PRD viven en docs/prd/ dentro del proyecto. Hay una plantilla lista en docs/prd/_template.md — Claude la usa de molde y deja un archivo por funcionalidad.

2 · Elegí el modelo adecuado

dentro de Claude · cambiar de modelo
> /model        # abre el selector y elegís de una lista
  • Para pensar y escribir el PRD → un modelo más analítico (p. ej. Opus). Razona mejor las decisiones.
  • Para construir la funcionalidad → un modelo equilibrado y veloz (p. ej. Sonnet). Va más rápido en el día a día.
🔁

No es obligatorio, pero ayuda: escribís el PRD con el modelo “pensante”, hacés /model para pasar al rápido y recién ahí pedís que lo construya.

3 · Pedí la construcción

Con el PRD listo, una instrucción del proyecto (/ship-feature) dispara todo el flujo: Claude lo construye, lo prueba y te deja una propuesta de cambio para aprobar.

  1. Describís qué querés (o le decís “seguí el PRD de tutores”).
  2. Claude lo construye siguiendo el estilo del proyecto.
  3. Lo prueba solo (revisa que no rompa nada).
  4. Te deja un “PR” — una propuesta de cambio en GitHub que alguien del equipo aprueba antes de que sea oficial.

Abrí Claude Code en la carpeta del repo y escribí:

dentro de Claude · en la carpeta del repo
> /ship-feature pantalla de tutores (dueños de las mascotas)
                con alta, edición y baja, igual que la de pacientes

Si preferís no acordarte del comando, alcanza con pedirlo en palabras:

💬 Pegale esto a Claude

Quiero sumar a la app una pantalla para gestionar tutores (los dueños de las mascotas): poder verlos en una lista, agregar uno nuevo, editarlo y borrarlo. Hacelo siguiendo el estilo de la pantalla de pacientes y dejame el cambio listo para revisar.

Para que salga bien, pedilo así

  • Una funcionalidad por vez. “Pantalla de tutores”, no “rehacé toda la app”.
  • Decí a qué se parece. “Igual que pacientes” le da a Claude el molde a copiar.
  • Aclará qué tiene que hacer el usuario. Ver, agregar, editar, borrar, buscar…
  • Probalo vos al final. Pedile “mostrame cómo quedó en la app” y revisá en el navegador antes de aprobar.
✅

Antes de dejarte el cambio, Claude corre las pruebas del proyecto. Si algo falla, lo arregla y recién ahí te avisa. Vos aprobás el PR cuando estés conforme.
