01

¿Qué hace este proyecto?

MascotApp es una plataforma de historia clínica veterinaria: un sistema donde una clínica registra tutores (dueños), sus mascotas y cada visita médica con diagnóstico y tratamiento. Hay dos roles de usuario: ADMIN y VET, con autenticación JWT (access token de 15 min + refresh de 7 días).

02

Arquitectura del monorepo

Es un monorepo con npm workspaces: dos aplicaciones bajo apps/ que se desarrollan, testean y deployan juntas.

apps/web
React 18 · Vite · shadcn/ui
React Query · React Router :5173
⟶HTTP · JWT
apps/api
NestJS 10 · Passport JWT
Helmet · Throttler · Zod env :3001 /api
⟶Prisma 5
PostgreSQL 16
Docker · postgres:16-alpine
migraciones + seed :5432
backend

🦴apps/api

Módulos por dominio (auth, users, tutors, pets, visits) + common/ con guards, filtros y pipes globales. Validación de env con Zod al arrancar: si falta una variable, no inicia. Tests con Jest.

frontend

🐈apps/web

Organización por features (features/auth, features/dashboard), primitivas shadcn/ui en components/ui, tema con CSS variables en theme/tokens.css. Tests con Vitest + Playwright para e2e.

seguridad

🔐Auth & guards

JWT HS256 con JwtAuthGuard global — los endpoints públicos se marcan con @Public(). Autorización por rol con @Roles() + RolesGuard. Rate limit: 100 req/60s.

infra

🐳Docker

Builds multi-stage (node:20-alpine; el web se sirve con nginx). docker-compose.yml levanta db + api + web, y corre prisma migrate deploy automáticamente antes de iniciar la API.

03

Configuración requerida

Requisitos previos

  • Node ≥ 20 — hay un .nvmrc, así que alcanza con nvm use.
  • npm — el monorepo usa npm workspaces (no pnpm ni yarn).
  • Docker + Docker Compose — al menos para PostgreSQL.

Archivos .env

Hay tres archivos .env.example que copiar. El de la raíz alimenta a docker-compose; los de cada app se usan para desarrollo local:

terminal — copiar los tres .env
$ cp .env.example .env
$ cp apps/api/.env.example apps/api/.env
$ cp apps/web/.env.example apps/web/.env

$ openssl rand -base64 48
⚠️

06

Cómo crear una feature

1

Partí de master actualizado y creá tu rama

Convención sugerida (no hay una documentada todavía): feat/, fix/, chore/ + descripción corta en kebab-case.

terminal
$ git checkout master && git pull
$ git checkout -b feat/mi-feature
2

Backend: seguí el patrón de los módulos existentes

Cada dominio es un módulo NestJS autocontenido. Si agregás, por ejemplo, "vacunas":

apps/api/src/vaccines/
vaccines/
├── vaccines.module.ts        # registrarlo en app.module.ts
├── vaccines.controller.ts    # @Roles() / @Public() según corresponda
├── vaccines.service.ts       # inyecta PrismaService
└── dto/                      # class-validator (whitelist estricta)
  • ¿Tocás el modelo de datos? Editá prisma/schema.prisma → npm run prisma:migrate → npm run prisma:generate. Commiteá la migración.
  • Los DTOs usan class-validator; el ValidationPipe global rechaza campos no whitelisted.
  • Decorá los endpoints con Swagger (@ApiTags, @ApiOperation) como en los módulos existentes.
3

Frontend: una carpeta por feature

Espejá el patrón de features/auth y features/dashboard:

apps/web/src/features/vaccines/
features/vaccines/
├── VaccinesPage.tsx          # registrar la ruta en routes/router.tsx
├── components/               # componentes propios de la feature
└── hooks/useVaccines.ts      # React Query contra lib/api
  • Usá las primitivas de components/ui (shadcn) en vez de inventar estilos nuevos.
  • Llamadas HTTP a través del cliente de lib/ + React Query; nada de fetch suelto.
  • Formularios con react-hook-form + zod, como el login.
  • Colores y espaciado salen de theme/tokens.css — no hardcodees hex.
4

Tests + calidad, antes de commitear

No hay CI todavía, así que esta pasada local es el gate de calidad:

el "pre-flight" completo
$ npm run lint        # 0 warnings o falla
$ npm run typecheck
$ npm run test
$ npm run test:e2e    # si tocaste flujos de UI
💡

Regla de oro: antes de escribir desde cero, abrí el módulo/feature más parecido (por ejemplo pets) y copiá su estructura. La consistencia entre módulos es deliberada.

07

Cómo crear un PR

feat/mi-feature→ commits convencionales→ push→ PR → master→ review & merge

Commits: Conventional Commits

El historial ya usa este formato (chore: scaffold MascotApp monorepo…). Mantenelo:

ejemplos
feat(api): add vaccines module with CRUD endpoints
fix(web): handle expired refresh token on dashboard
chore: bump prisma to 5.21
test(api): cover visits service edge cases

Abrir el PR con GitHub CLI

terminal
$ git push -u origin feat/mi-feature
$ gh pr create --base master \
    --title "feat(api): add vaccines module" \
    --body  "## Qué cambia ... ## Cómo probar ..."

Checklist antes de pedir review

🚨

Nunca pushees directo a master ni commitees credenciales reales. Los usuarios seed y el password de Postgres del compose son solo para desarrollo local.

📌

Pendientes del repo (buenas primeras contribuciones): no hay CI/CD (GitHub Actions), ni PR template, ni CONTRIBUTING.md, ni hooks de pre-commit activos.
